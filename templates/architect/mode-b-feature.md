# モード B: 機能設計モード — 詳細フロー (5 ステップ)

モード B は **1 機能あたり 2 つのドキュメント** を生成します:
- `docs/requirements/<feature-name>.md` — この機能の要件定義書（何を作る・なぜ作る）
- `docs/specs/<feature-name>.md` — この機能の詳細仕様書（どう動く・受け入れ基準）

両方を同じセッションで連続生成し、ユーザー承認を取りながら進めます。

## B-1: ヒアリング（8 項目・1 問ずつ）

**質問は 1 つずつ投げかけて、ユーザーの回答を受けてから次に進む**こと。8 項目を 1 メッセージにまとめて聞かない。

#### 進め方の基本ルール

- **1 ターン = 1 質問**: 各メッセージで聞くのは 1 つ。回答が来てから次へ。
- **重複質問は禁止**: ユーザーが既に話してくれた内容を再質問しない。1 回の回答で複数項目が埋まったら、その分はスキップして次の未確認項目に進む。
- **会話の継続を尊重**: ユーザーが同じ話題を膨らませている（仕様の補足・周辺機能の話・背景の説明など）と判断したら、**こちらから次の質問を割り込ませず、相づちと深掘りで聞き続ける**。話題が一段落してから次の質問へ。
- **不明・暫定OK**: 「不明」「不要」「未定」も有効な回答。記録だけしておく。

#### 最初の問いかけ

```
【機能設計のヒアリングを始めます!】
仕様書を作るために、いくつか聞いていきますね!1 つずつ進めるので気軽に答えてください。
分からない項目は「不明」「未定」で全然OKですよ〜。

まず最初に — この機能で **何を達成したい** ですか?(1〜2 文で)
```

#### 以降の質問項目（順序は柔軟、未確認のものを 1 つずつ）

1. 目的: 何を達成したい?
2. ユーザー: 誰が使う?(一般ユーザー / 管理者 / バッチ 等)
3. トリガー: いつ・どうやって発動?(ボタン押下 / 定期実行 / API 呼び出し 等)
4. 入力: 何を受け取る?(項目名・型・必須/任意)
5. 出力: 何を返す/表示する?(成功時・失敗時)
6. 成功条件: 何が起きたら "うまくいった" と言える?
7. 制約: 壊してはいけないもの / 性能要件 / セキュリティ要件
8. 範囲外: 今回は意図的にやらないこと

#### アプリ構想モードからの遷移時

vision.md と roadmap.md の内容を使って **8 項目のうち推測できるものを暫定値で先に埋めて提示** し、その後 1 項目ずつ確認していく:

```
roadmap.md と vision.md から、以下が暫定値として推測できました:

- 目的: <暫定値>
- ユーザー: <暫定値>
- ...

この暫定値からスタートします。1 つずつ確認していきますね。
まず「目的」: <暫定値> で合っていますか?(違うなら正しい内容を教えてください)
```

未確認の項目（暫定値が無い項目）も 1 問ずつ聞く。

#### 全項目が出揃ったら

要約して確認:

```
【ここまでの理解】
- 目的: ...
- ユーザー: ...
- トリガー: ...
- 入力: ...
- 出力: ...
- 成功条件: ...
- 制約: ...
- 範囲外: ...

以上で合っていますか? 補足や違うところがあれば教えてください。
```

承認が出るまで B-2 に進まない。

## B-2: 粒度判定（Small / Medium / Large）

- **Small** = 1 ファイルで完結 / 想定実装 < 50 行 / テストケース ≤ 3 → 続行
- **Medium** = 2〜5 ファイル / 想定実装 50〜300 行 / テストケース 4〜10 → 続行
- **Large** = 6 ファイル以上 / 300 行超 / テストケース 11 件以上 → **分割提案が必須**

Large のときの出力例:
```
【粒度判定: Large】
このまま 1 つの仕様書にすると肥大化します。以下に分割することを提案します:

- feat A: <子機能 A> (Medium)
- feat B: <子機能 B> (Small)
- feat C: <子機能 C> (Medium)

この分割で進めますか? それともこのまま 1 仕様書で行きますか?
```

ユーザー承認後、分割した場合は子機能ごとに B-3 を繰り返す。

## B-3: 要件定義書（機能ごと）の生成 — **セクションごとに対話形式で進める**

`docs/requirements/_template.md` を読み込み、その章立てに従って `docs/requirements/<feature-name>.md` を作成。テンプレートが無ければ「`docs/requirements/_template.md` が見つかりません。先に `/cc-development-team:init-dept` を実行してください」と報告して止まる。

**プロジェクト全体のドキュメントとの整合性を保つ**:

書き始める前に以下を `Read` で確認し、矛盾しないように記述する:

- `docs/basic-design/basic-design.md` (基本設計書): 機能一覧から該当 F-XXX を引いてこのファイル冒頭に記載。全体方針 (認証認可・非機能設計) と矛盾しないよう。

basic-design.md が存在しない場合（develop モードから直接呼ばれた等）は、その旨を変更ログに記録して、要件定義書単独で完結する形で書き出す。

#### B-3-1: 章立ての確認とドラフト方針の合意

`docs/requirements/_template.md` の章一覧をユーザーに提示し、進め方を伝える:

```
【要件定義書をセクションごとに作っていきますね!】
ファイル: docs/requirements/<feature-name>.md

これから以下の章を 1 つずつ作っていきます。各章ごとに「こう書きました、これで OK?」と
確認しますので、その都度修正や追加要望を伝えてください!

順番:
1. <章タイトル 1>
2. <章タイトル 2>
3. ...

最初の章 (<章タイトル 1>) から始めてもいいですか?
```

ユーザー承認が出てから B-3-2 へ。

#### B-3-2: セクションごとのドラフト → 確認 → 次へ

**最優先ルール (推測禁止):**
- **ヒアリング (B-1) で明示されていない情報をドラフトに含めない**。書きたい内容が未確認なら、まずユーザーに質問する
- 「こうだろう」「一般的にはこう」という推測で埋めない
- basic-design.md / vision.md / requirements の既存ドキュメントから **明示的に引用できる情報のみ** 流用してよい
- 不足項目があれば「この章を書くために `<項目名>` を教えてください」と **章ごとに先にヒアリング**

各章について以下を **1 章ずつ** 繰り返す:

1. **章を書くために必要な情報をユーザーに確認**:
   - B-1 のヒアリング結果と既存ドキュメントから引用できる項目を洗い出す
   - 不足項目があれば、その章に着手する前に質問:
     ```
     【<章タイトル> を書く前に確認したいこと】
     この章を埋めるのに以下が必要です。教えてください:

     - <項目 1>: ?
     - <項目 2>: ?

     「分からない」「未定」「飛ばして」も有効な回答です (その場合は「未確定」として残します)。
     ```
   - ユーザーの回答を待ってから次のステップへ
2. **ドラフト案を会話内に提示** する (まだファイルには書かない)。ユーザーから得た情報と既存ドキュメントの **明示的な引用のみ** で章の中身を Markdown 形式で示す。
3. **以下のフォーマットで承認を求める**:
   ```
   【<章タイトル> の案です】

   <章のドラフト Markdown>

   ---
   この内容で OK ですか?
   - OK → 次の章 (<次の章タイトル>) に進みます
   - 修正したい → どこを直すか教えてください (該当箇所だけ書き換えます)
   - 一旦止めたい → 「ストップ」と伝えてください
   ```
4. **OK が出るまで次の章には進まない**。修正要望が出たらその場で反映し、再提示。
5. ハッピーパスと代替（エラー）シナリオは両方カバーするが、いずれもユーザーが言及したシナリオに限定する。未言及のシナリオを推測で追加せず、「他に想定しておきたいシナリオはありますか?」と必ず確認。
6. 不明な項目は「未確定」として残す。推測で埋めず、ユーザーに「ここはどうしますか?」と質問。

進め方の注意:
- **1 メッセージで複数章を投げない**（情報過多になる）
- ユーザーが「全部 OK で進めて」と言った場合のみ、まとめて残りの章を提示してよい (ただし上記の最優先ルールは維持。不足情報があればまとめ提示前にヒアリング)
- 章のドラフトが長くなる場合は、章内をさらに小ブロックに分けて確認してもよい

#### B-3-3: 全章承認後、ファイルに書き出す

全章について承認が取れたら、**承認済みの内容をそのまま** `docs/requirements/<feature-name>.md` に `Write` する。会話で承認していない内容を勝手に追加しないこと。

書き出し完了をユーザーに伝える:

```
【要件定義書 (<feature-name>) を確定しました!】
ファイル: docs/requirements/<feature-name>.md

各章で確認した内容をそのまま書き出しています。
次は詳細仕様書 (B-4) に進みます。
```

ユーザーが「やっぱ全体見て直したい」と言った場合は、該当箇所だけ Edit で更新する（書き直しは最小差分で）。

## B-4: 詳細仕様書（機能ごと）の生成 — **セクションごとに対話形式で進める**

`docs/specs/_template.md` を読み込み、その章立てに従って `docs/specs/<feature-name>.md` を作成。テンプレートが無い場合は「`docs/specs/_template.md` が見つかりません。先に `/cc-development-team:init-dept` を実行してください」と報告して止まる。

**入力情報:**

書き始める前に以下を `Read` で確認:

- `docs/requirements/<feature-name>.md` (B-3 で作成した要件定義書): 機能要件・操作シナリオ・非機能要件を実現する具体仕様に落とす
- `docs/basic-design/basic-design.md` (基本設計書): 採用技術スタック・データモデル・認証認可方針と整合させる

仕様書冒頭の **機能 ID** 欄に、basic-design.md の F-XXX を記入する。

basic-design.md / requirements/<feature-name>.md が存在しない場合（develop モードから直接呼ばれた等）は、その旨を変更ログに記録して、仕様書単独で完結する形で書き出す。

#### B-4-1: 章立ての確認とドラフト方針の合意

`docs/specs/_template.md` の章一覧をユーザーに提示し、進め方を伝える:

```
【詳細仕様書もセクションごとに作っていきます!】
ファイル: docs/specs/<feature-name>.md

要件定義書をベースに、こちらも 1 章ずつ提示→確認の流れで進めますね。
特に **受け入れ基準は Given-When-Then 形式** で書くので、ケース漏れがないか
一緒に確認していきましょう!

順番:
1. <章タイトル 1>
2. <章タイトル 2>
3. ...

最初の章 (<章タイトル 1>) から始めてもいいですか?
```

ユーザー承認が出てから B-4-2 へ。

#### B-4-2: セクションごとのドラフト → 確認 → 次へ

**最優先ルール (推測禁止):**
- **要件定義書 (B-3 で確定) や basic-design.md に明示されていない情報をドラフトに含めない**。書きたい内容が未確認なら、まずユーザーに質問する
- 「こうだろう」「一般的にはこう」「業界標準ではこう」という推測で埋めない
- API シグネチャ・データモデル・画面レイアウト・内部ロジックなど **技術判断が必要な箇所** こそ、こちらから選択肢を 2〜3 個提示してユーザーに選んでもらう (勝手に決めない)
- 受け入れ基準のケースは **ユーザーが言及したもの + ユーザーが追加を承認したものに限定**。未承認のケースを推測で追加しない

各章について以下を **1 章ずつ** 繰り返す:

1. **章を書くために必要な情報をユーザーに確認**:
   - 既存ドキュメント (`docs/requirements/<feature-name>.md`、`docs/basic-design/basic-design.md`) から **明示的に引用できる情報** を洗い出す
   - 不足分は質問してから着手:
     ```
     【<章タイトル> を書く前に確認したいこと】
     この章を埋めるのに以下が必要です:

     - <項目 1>: ?
     - <項目 2>: 選択肢を 2〜3 個提示する場合
       a) <候補 A>
       b) <候補 B>
       c) その他 (自分で指定)

     「分からない」「未定」も有効な回答です。
     ```
2. **ドラフト案を会話内に提示** する (まだファイルには書かない)。ユーザーから得た情報と既存ドキュメントの **明示的な引用のみ** で章の中身を Markdown 形式で示す。
3. **以下のフォーマットで承認を求める**:
   ```
   【<章タイトル> の案です】

   <章のドラフト Markdown>

   ---
   この内容で OK ですか?
   - OK → 次の章 (<次の章タイトル>) に進みます
   - 修正したい → どこを直すか教えてください
   - ケース追加したい → 想定漏れがあれば言ってください (受け入れ基準で重要)
   ```
4. **OK が出るまで次の章には進まない**。
5. **「受け入れ基準」章では特に丁寧に**: ハッピーパス・エラー系・境界値ケースを順番に提示し、それぞれ Given-When-Then 形式で確認。**ユーザーから明示されていないケースは出さず**、「他にハッピーパス / エラー系 / 境界値のケースはありますか?」と必ず聞く。
   ```
   【受け入れ基準 (ハッピーパス)】
   - Given: <前提条件>
   - When: <操作 / イベント>
   - Then: <期待結果>

   このケースで OK?他のハッピーパスは?
   ```
   その後「エラー系」「境界値」も同様に、毎回「他にありますか?」を聞く。
6. 曖昧表現（「ちゃんと動く」「適切に処理する」）は禁止。具体的な観測可能事象で書く。
7. 不明な項目は「未確定」として残す。推測で埋めず、ユーザーに「ここはどうしますか?」と質問。

進め方の注意:
- **1 メッセージで複数章を投げない**
- ユーザーが「全部 OK で進めて」と言った場合のみ、まとめて残りを提示してよい (ただし上記の最優先ルールは維持。不足情報があればまとめ提示前にヒアリング)
- 技術判断が必要な章では **必ず選択肢を 2〜3 個提示**。提示した選択肢以外の候補も「これら以外でやりたい構成がありますか?」と必ず聞く

#### B-4-3: 全章承認後、ファイルに書き出す

全章について承認が取れたら、**承認済みの内容をそのまま** `docs/specs/<feature-name>.md` に `Write` する。会話で承認していない内容を勝手に追加しないこと。

書き出し完了をユーザーに伝える:

```
【詳細仕様書 (<feature-name>) を確定しました!】
ファイル: docs/specs/<feature-name>.md

各章で確認した内容をそのまま書き出しています。
次は手動操作タスクの確認 (必要時のみ) → B-5 (最終確認) に進みます。
```

#### B-4-extra: 手動操作タスクの抽出（必要時のみ）

ヒアリング・仕様策定の過程で **「人間が UI などで外部サービスを設定しないと完成しない」項目** が見つかったら、以下を実施する:

該当例:
- Firebase / Supabase などの BaaS プロジェクト作成
- OAuth プロバイダ (Google / GitHub / LINE 等) のクライアント ID 発行
- ドメイン取得・DNS 設定
- App Store Connect / Google Play Console の準備
- API キー登録、課金設定、証明書発行 等

手順:
1. `docs/manual-tasks/<feature-name>.md` を `docs/manual-tasks/_template.md` の章立てに従って作成（既存があれば追記）
2. 各タスクに「カテゴリ」「ステータス (作成時点では未着手)」「手順」「完了の確認方法」「ブロッカー」を埋める
3. 作成した仕様書 `docs/specs/<feature-name>.md` の「手動操作タスク」セクションから、この手動タスクファイルへリンクを張る

手動タスクが **無い場合** は、仕様書側の「手動操作タスク」セクションに「無し」と書いて、`docs/manual-tasks/<feature-name>.md` は作らなくてよい。

## B-5: 全体確認 → 確定

B-3 / B-4 で各章ごとに承認済みのはずなので、ここでは **全体を通して気になる点が無いかの最終確認** だけ取る。

```
【設計完了の最終確認です!】
今回作成したドキュメント:
- 要件定義書: docs/requirements/<feature-name>.md
- 詳細仕様書: docs/specs/<feature-name>.md
- 手動操作タスク: docs/manual-tasks/<feature-name>.md (該当あれば)

全章 OK が出ているので、ステータスを「確定」にして締めようと思います。
何か全体を通して気になる点はありますか?(無ければそのまま確定します!)
```

ユーザーが「OK」または無修正で進めると判断したら、**両方のドキュメントの末尾** に `## ステータス: 確定 (YYYY-MM-DD)` を追記。

修正要望が出たら該当箇所を `Edit` で更新し（最小差分）、再度この最終確認に戻る。

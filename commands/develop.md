「$ARGUMENTS」

---

このコマンドは **「develop モード」に入る** ためのトリガーです。一度入ったら、ユーザーが明示的に終了するまで、ユーザーの後続メッセージを **すべて開発依頼として処理** し続けてください。

## モード開始時のアナウンス

最初に必ず以下のメッセージをユーザーに表示してください:

```
【develop モードに入りました!行きましょう〜!】

ここから先は、あなたの依頼を 1 件ずつ処理していきます!
1 件の処理が終わると、また次の依頼を聞きますね。

- 終了するには: 「終了」「exit」「終わり」「もう大丈夫」などと伝えてください
- 別モードに切り替えるには: `/cc-development-team:architect` などのスラッシュコマンドを実行（モード自動終了）
```

## サイクル範囲のヒアリング (`$CYCLE_MODE`)

続けて、1 サイクルでどこまで自動実行するかを必ずユーザーに尋ねる:

```
【1 サイクルの範囲を選んでください】
番号で答えてくださいね!

a) フルサイクル (実装 → レビュー → テスト → 仕様書更新 → 動作確認)
   ※ tester も毎回呼んで、テストも合わせて回します
b) 実装重視 (実装 → レビュー → 仕様書更新 → 動作確認)
   ※ tester は呼ばず実装に集中。テストは別途 /cc-development-team:test で実行
```

ユーザーの回答を以下のように `$CYCLE_MODE` として保持:

- a を選択 → `$CYCLE_MODE = "full"`
- b を選択 → `$CYCLE_MODE = "lean"`

途中での切り替えにも対応する。次のいずれかの発言があれば `$CYCLE_MODE` を更新し、その旨をユーザーに 1 行で確認:

- 「フルに切り替えて」「テストも入れて」 → `$CYCLE_MODE = "full"`
- 「実装重視に切り替えて」「テスト外して」「軽量で」 → `$CYCLE_MODE = "lean"`

## 初回の依頼を受け取る

`$ARGUMENTS` の内容で分岐:

- **`$ARGUMENTS` に値がある場合**: それを最初の依頼とみなして「1 件の処理サイクル」に進む
- **`$ARGUMENTS` が空の場合**: ユーザーに以下を尋ねる:

  ```
  最初の依頼を教えてください!
  「○○機能を作りたい」「△△のバグを直したい」「□□をリファクタしたい」など、自由に書いてOKです〜
  ```

  ユーザーの回答を受けて「1 件の処理サイクル」に進む。

## 1 件の処理サイクル

ユーザーから依頼を受け取るたびに、以下を実行してください。

### Step 0: feature-name の決定 & 仕様書の有無確認

1. ユーザーの依頼内容から **kebab-case の feature-name** を決定する（例: "ログインフォーム作りたい" → `login-form`）。決めた名前をユーザーに見せて確認:
   ```
   今回の作業を `<feature-name>` として扱います。よろしいですか?
   (別の名前にしたい場合は教えてください)
   ```
   確定したら、その名前を `$CURRENT_FEATURE` として保持してください。

2. `docs/specs/$CURRENT_FEATURE.md` の **存在を確認** する（`Read` で読めるか試す）:
   - 読める → `$SPEC_EXISTS = true`（既存仕様書あり）
   - 読めない/無い → `$SPEC_EXISTS = false`（仕様書なし、後で back-fill 必要）

3. ユーザーに状態を伝える:
   - `$SPEC_EXISTS = true` の場合:
     ```
     仕様書 `docs/specs/$CURRENT_FEATURE.md` を見つけました。
     これに沿って実装します。
     ```
   - `$SPEC_EXISTS = false` の場合:
     ```
     仕様書はまだありません。実装が終わった後に仕様書を逆生成します。
     ```

### Step 1: 実装 (developer)

`developer` サブエージェントを起動し、`$SPEC_EXISTS` の値で分岐して以下の指示を渡す:

#### `$SPEC_EXISTS = true` の場合
> 機能 `$CURRENT_FEATURE` を、`docs/specs/$CURRENT_FEATURE.md` の仕様書に **沿って実装** してください。CLAUDE.md と既存慣習を守り、最小差分で進めてください。
>
> もし実装中に仕様書に記載のない機能要件・エッジケース・暗黙の前提が出てきた場合は、それを **「仕様書外で追加した内容」として明示的に変更ログに記録** してください（後のフェーズで仕様書を更新するため）。
>
> **UI / フロントエンドのコード** を含む場合は、実装に入る前に必ずプロジェクト種別に応じた **Frontend Skill** を呼び出してください（Web → `everything-claude-code:frontend-patterns`、iOS → `everything-claude-code:swiftui-patterns` + 必要に応じて `liquid-glass-design`、Android/KMP → `everything-claude-code:compose-multiplatform-patterns` 等。詳細は agents/developer.md の「UI / フロントエンド作業時の Frontend Skills 活用」セクション参照）。

#### `$SPEC_EXISTS = false` の場合
> 機能 `$CURRENT_FEATURE` をユーザーの依頼内容（直近のユーザーメッセージ参照）に基づいて実装してください。仕様書 `docs/specs/$CURRENT_FEATURE.md` は **まだありません**。完了時に back-fill するため、変更概要・影響範囲・暗黙の受け入れ基準を箇条書きで変更ログとして残してください。CLAUDE.md と既存慣習に沿い、最小差分で進めてください。
>
> **UI / フロントエンドのコード** を含む場合は、実装に入る前に必ずプロジェクト種別に応じた **Frontend Skill** を呼び出してください（同上）。

完了後、developer が出力した「変更ログ」と「仕様書外で追加した内容（あれば）」を保持してください（Step 4 で使用）。

### Step 2: レビュー (reviewer)

`reviewer` サブエージェントを起動し、次の指示を渡す:

> 実装差分の品質・セキュリティ・既存慣習との整合性を点検してください。`$SPEC_EXISTS = true` の場合は **仕様書との整合性** も併せて確認してください。severity 付きで指摘し、最終判定 (Approve / Request Changes / Block) を出してください。

`Request Changes` 以上の指摘があれば `developer` に修正依頼を出し、Step 2 を **Approve が出るまで反復** してください。

### Step 3: テスト作成 + 全体実行 (tester) — `$CYCLE_MODE = "full"` のときのみ

`$CYCLE_MODE = "lean"` の場合は **Step 3 全体をスキップして Step 4 へ進む**。

`$CYCLE_MODE = "full"` の場合のみ、`tester` サブエージェントを起動し、次の指示を渡す:

> 機能 `$CURRENT_FEATURE` の実装に対応する **テストを新規作成** してください。
> - `docs/specs/$CURRENT_FEATURE.md` が存在する場合: 受け入れ基準を Given-When-Then の形でテストに翻訳
> - 存在しない場合: developer の変更ログと実装内容をベースに、適切なレイヤ（ユニット / 統合 / E2E）でテストを書く
>
> テスト作成後、**プロジェクト全体のテストランナーを実行** し、新規テスト＋既存テストが **全件グリーン** であることを確認してください。失敗時の対応:
> - 実装側のバグが原因 → このコマンドに差し戻し（ユーザー承認を経て `developer` に修正依頼します）
> - テスト側の問題 → 自分で修正
> - 既存テストが壊れた → 原因を特定してレポート（実装の副作用かテストの不備か）

`tester` から失敗 or 改善候補が報告された場合、ユーザーに以下を尋ねる:

```
【テスト結果に問題があります】
失敗 N 件 / 改善候補 M 件

修正に進みますか?
a) 失敗テストを developer に修正させる
b) このまま次のステップへ進む（修正は後で /cc-development-team:test で対応）
```

`a` を選ばれた場合は developer 修正 → tester 再実行を **最大 3 反復**。3 回失敗で立ち止まりユーザーに状況報告。
`b` を選ばれた場合は失敗を残したまま Step 4 へ進む（完了報告でその旨を明示）。

### Step 4: 仕様書の新規作成 / 更新（必要時のみ）

以下の条件で分岐します。**該当しない場合は Step 4 を完全にスキップ** してください。

#### ケース A: `$SPEC_EXISTS = false` — 新規 back-fill

`architect` サブエージェントを起動し、次の指示を渡す:

> developer が実装した内容から、仕様書 `docs/specs/$CURRENT_FEATURE.md` を **新規生成** してください。実際の変更ファイル・コード・developer の変更ログを根拠に、`docs/specs/_template.md` の章立てに従って書き出してください。受け入れ基準は Given-When-Then 形式。末尾に `## ステータス: back-fill (YYYY-MM-DD)` を追記してください。

#### ケース B: `$SPEC_EXISTS = true` かつ「仕様書外で追加した内容」がある

`architect` サブエージェントを起動し、次の指示を渡す:

> 既存の仕様書 `docs/specs/$CURRENT_FEATURE.md` を、developer が今回追加した「仕様書外の内容」を反映する形で **更新** してください。元のテキストを破壊せず、差分は **改訂履歴セクション** として末尾に追記する形で記録してください。受け入れ基準が増えた場合は Given-When-Then 形式で。末尾に `## ステータス: 更新 (YYYY-MM-DD)` を追記してください。
>
> 仕様書外で追加された内容（developer の変更ログから）:
> - <差分 1>
> - <差分 2>

#### ケース C: `$SPEC_EXISTS = true` かつ「仕様書外で追加した内容」が無い → **スキップ**

仕様書と実装が一致しているので、architect は呼ばない。

### Step 5: モック起動の確認（Web / Mobile プロジェクトのみ）

実装したものを **動かして確認するか** をユーザーに尋ねる。

#### 5-1. プロジェクト種別と起動コマンドの判定

`docs/CLAUDE.md` の「プロジェクト種別」と `dept/developer/CLAUDE.md` の「ビルド・実行コマンド」を `Read` で取得する。
それでも不明なら、以下のヒントから推定:

- **Web**: `package.json` (`scripts.dev` or `start`) / `manage.py` (Django) / `Gemfile` (Rails) / `go.mod` (Go) など
- **iOS**: `*.xcodeproj` または `Package.swift`
- **Android**: `build.gradle.kts` または `settings.gradle.kts`
- **Flutter**: `pubspec.yaml`
- **React Native**: `react-native.config.js` または `app.json`
- **その他（CLI、ライブラリ等）**: そもそも起動する対象が無いので Step 5 全体をスキップ

#### 5-2. ユーザーに確認

以下のフォーマットで尋ねる（`$CYCLE_MODE` に応じて文言を切り替え）:

`$CYCLE_MODE = "full"` のとき:
```
【動作確認】
実装とテストが完了しました!実際に起動して動作確認しますか?
```

`$CYCLE_MODE = "lean"` のとき:
```
【動作確認】
実装が完了しました!実際に起動して動作確認しますか?
(テストは別途 /cc-development-team:test で実行できます)

- プロジェクト種別: <Web / iOS / Android / Flutter / 等>
- 推定起動コマンド: <コマンド>
- 起動先: <localhost:3000 / iOS シミュレータ (iPhone 15) / Android Emulator / 等>

起動しますか? (yes / no / 別のコマンドで)
```

#### 5-3. ユーザー回答による分岐

##### `yes` の場合

**起動前に必ず最終確認**:

```
以下のコマンドを実行します。OK ですか?
  <コマンド>

(注意: dev server / シミュレータはバックグラウンドで動き続けます。停止方法は後ほどお伝えします)
```

確認が取れたら、以下のルールで実行:

- **Web の場合**:
  - `Bash` を `run_in_background: true` で実行
  - 数秒待ってから、ログを確認してアクセス URL を抽出（"localhost:3000" 等）
  - ユーザーに `「http://localhost:XXXX で起動しました。停止するときは別ターミナルで `kill <PID>` または該当プロセスを Ctrl+C」` と伝える

- **iOS の場合**:
  - `xcrun simctl boot <デバイス名>` (例: `xcrun simctl boot 'iPhone 15'`) をフォアグラウンドで実行
  - `xcrun simctl list devices` で利用可能デバイスを確認するのが安全
  - `xcodebuild` でビルド + シミュレータインストール、または Xcode を `open` で起動する案内
  - シミュレータの起動完了を待ってから、`xcrun simctl launch booted <bundle-id>` でアプリを起動

- **Android の場合**:
  - `emulator -list-avds` で利用可能なエミュレータ確認
  - `emulator -avd <名前>` をバックグラウンドで起動
  - 起動完了を `adb wait-for-device` で待ってから、`./gradlew installDebug` でビルド+インストール
  - `adb shell am start -n <package>/<activity>` でアプリ起動

- **Flutter の場合**:
  - `flutter run` を `run_in_background: true` で実行
  - デバイス選択が必要なら事前に `flutter devices` で確認

- **React Native の場合**:
  - `npx react-native run-ios` または `run-android` を `run_in_background: true` で実行

##### `no` の場合

「起動はスキップしました。後で確認する場合は手動で `<コマンド>` を実行してください」と伝えて Step 6 へ。

##### `別のコマンドで` の場合

ユーザーに「どのコマンドで起動しますか?」と尋ね、回答に従って同様に実行確認 → 実行。

#### 5-4. 起動後の取り扱い

- 起動したプロセスの PID または識別情報を保持し、ユーザーに伝える
- develop モードのサイクルを続ける場合、次の依頼の処理は通常通り継続（dev server はそのまま動かしておく）
- HMR (Hot Module Reload) があれば修正は自動反映される旨を伝える
- 大きな修正後は再起動が必要かもしれないことを案内

### Step 6: このサイクルの完了報告

以下を簡潔に報告:

1. 今回の `$CURRENT_FEATURE`
2. サイクル範囲: `$CYCLE_MODE`（full / lean）
3. 仕様書の状態: 「新規作成」/「既存あり更新」/「既存ありそのまま (変更不要)」のいずれか + パス
4. 変更したファイル一覧
5. **テスト状況**:
   - `$CYCLE_MODE = "full"` の場合: 新規追加テスト件数 + 合格 / 失敗 / スキップ件数。未解決の失敗が残っていれば明示
   - `$CYCLE_MODE = "lean"` の場合: 「このサイクルではテストを実行していません。`/cc-development-team:test $CURRENT_FEATURE` で実行できます」と案内
6. **手動操作タスクの有無**:
   - developer が `docs/manual-tasks/$CURRENT_FEATURE.md` を作成・更新した場合は、そのパスと **未完了タスク件数** を必ず明示する
   - 未完了タスクがある場合は「これらの設定が完了するまで <機能名> は完全には動作しません」と注意喚起
   - 該当無しなら「手動タスク: 無し」と明記
7. **起動状態**:
   - Step 5 で起動した場合: アクセス先 (URL / シミュレータ名) と停止方法
   - 起動しなかった場合: 「起動はスキップ」と一言
8. 残課題があれば箇条書き

## サイクル間の振る舞い（モード継続）

1 件の処理サイクルが終わったら、以下のメッセージを表示して **次の依頼を待つ** こと:

```
【1 件完了!お疲れさまでした!— develop モード継続中】

次の依頼はありますか?
- 次の機能 / 修正内容を教えてください!
- 終了する場合は「終了」「exit」「終わり」「もう大丈夫」などと伝えてくださいね
```

ユーザーから次のメッセージが来たら、その内容を **再び「依頼」として解釈** し、「1 件の処理サイクル」を Step 0 から実行してください。

---

## モード終了の判定

ユーザーのメッセージが以下のいずれかに該当する場合、develop モードを **終了** してください:

- 明示的な終了語句: 「終了」「exit」「終わり」「もう大丈夫」「やめる」「ストップ」「stop」「quit」
- 別モードへの切り替えを示唆: 「設計に戻りたい」「別のことしたい」「いったん休憩」
- 他のスラッシュコマンドの実行 (例: `/cc-development-team:architect`): この場合は自動的にモードが切り替わる

判断に迷う場合（曖昧な発言）は、必ずユーザーに確認:

```
develop モード中ですが、終了してよろしいですか?
- yes: モード終了
- no: 引き続き依頼をお聞きします!
```

## モード終了時の最終報告

モードを抜けるとき、以下をユーザーに報告してください:

1. **このセッションで処理した件数** と各 `$CURRENT_FEATURE` 名
2. 累計で変更したファイル数、追加したテスト件数（full サイクルがあった場合）
3. 仕様書の状態サマリ（新規作成 / 更新 / 変更なし の件数）
4. **テスト状況**: `lean` で進めた機能があれば「○○ はテスト未実行。`/cc-development-team:test` で実行してください」と案内
5. 「develop モードを終了しました」のアナウンス

---

## モード中の禁止事項

- ユーザーの依頼を受けたのに developer 起動をスキップする
- **`$CYCLE_MODE` を確認しないまま 1 サイクル目を回し始める**（モード開始時に必ずヒアリング）
- **`$CYCLE_MODE = "lean"` で tester を呼ぶ**（テストは `/cc-development-team:test` の責務）
- 「仕様書外で追加した内容」があるのに Step 4 をスキップ
- 「仕様書外で追加した内容」が無いのに architect を呼ぶ（無駄な呼び出し）
- **Step 5 の起動確認をユーザーに聞かずにスキップする**（CLI/ライブラリ等で起動対象が無い場合のみスキップ可）
- **ユーザー承認なしに dev server / シミュレータを勝手に起動する**（必ず最終確認）
- ユーザーが終了の意思を明示していないのに勝手にモードを抜ける
- develop モード中に勝手に `/cc-development-team:architect` 等を呼ぶ
- 1 件のサイクルが終わってないうちに「次の依頼を教えてください」と聞く

## 注意事項（モード永続化の仕組み）

このモードは **Claude の会話文脈** によって保持されます。以下のケースでは自動的にモードが失われる/中断される可能性があります:

- 長時間の会話で context が圧縮された場合
- 他のスラッシュコマンドを実行した場合
- Claude Code を再起動した場合

その際は、ユーザーが再度 `/cc-development-team:develop` を実行することでモードに戻れます。

「$ARGUMENTS」

---

このコマンドは **「develop モード」に入る** ためのトリガーです。一度入ったら、ユーザーが明示的に終了するまで、ユーザーの後続メッセージを **すべて開発依頼として処理** し続けてください。

## モード開始時のアナウンス

最初に必ず以下のメッセージをユーザーに表示してください:

```
【develop モードに入りました】

ここから先は、あなたの依頼を「開発先行ルート」で 1 件ずつ処理していきます。
1 件の処理が終わると、また次の依頼を聞きます。

- 1 件の流れ: developer (実装) → reviewer (レビュー) → architect (仕様書 back-fill) → tester (テスト)
- 終了するには: 「終了」「exit」「終わり」「もう大丈夫」などと伝えてください
- 別モードに切り替えるには: `/cc-development-team:design` などのスラッシュコマンドを実行（モード自動終了）
```

## 初回の依頼を受け取る

`$ARGUMENTS` の内容で分岐:

- **`$ARGUMENTS` に値がある場合**: それを最初の依頼とみなして、後述の「1 件の処理サイクル」に進む（feature-name を `$ARGUMENTS` から決定）
- **`$ARGUMENTS` が空の場合**: ユーザーに以下を尋ねる:

  ```
  最初の依頼を教えてください。
  「○○機能を作りたい」「△△のバグを直したい」「□□をリファクタしたい」など、
  自由に書いてください。
  ```

  ユーザーの回答を受けて、後述の「1 件の処理サイクル」に進む。

---

## 1 件の処理サイクル

ユーザーから依頼を受け取るたびに、以下を実行してください。

### 0. feature-name の決定

ユーザーの依頼内容から **kebab-case の feature-name** を決定する（例: "ログインフォーム作りたい" → `login-form`）。決めた名前をユーザーに見せて確認:

```
今回の作業を `<feature-name>` として扱います。よろしいですか?
(別の名前にしたい場合は教えてください)
```

確定したら、その名前を `$CURRENT_FEATURE` として保持してください。

### 1. 実装 (developer)

`developer` サブエージェントを起動し、次の指示を渡す:

> 機能 `$CURRENT_FEATURE` をユーザーの依頼内容（直近のユーザーメッセージ参照）に基づいて実装してください。仕様書 `docs/specs/$CURRENT_FEATURE.md` は **まだありません**。完了時に back-fill するため、変更概要・影響範囲・暗黙の受け入れ基準を箇条書きで残してください。CLAUDE.md と既存慣習に沿い、最小差分で進めてください。

### 2. レビュー (reviewer)

`reviewer` サブエージェントを起動し、次の指示を渡す:

> 実装差分の品質・セキュリティ・既存慣習との整合性を点検してください。仕様書はまだ存在しないので整合性チェックは飛ばし、コード自体の品質と現コードベースとの一貫性に集中してください。severity 付きで指摘し、最終判定を出してください。

`Request Changes` 以上があれば developer に差し戻し、Approve が出るまで繰り返す。

### 3. 仕様書 back-fill (architect) — **必ず実行**

`architect` サブエージェントを起動し、次の指示を渡す:

> developer が実装した内容から、仕様書 `docs/specs/$CURRENT_FEATURE.md` を **逆生成** してください。実際の変更ファイル・コード・developer の変更ログを根拠に、背景/目的・スコープ・要件・設計・受け入れ基準・リスクを書き起こしてください。既に `docs/specs/$CURRENT_FEATURE.md` が存在する場合は、改訂履歴を末尾に追記して内容を最新化してください。

**このフェーズはスキップ禁止です。**

### 4. テスト (tester)

`tester` サブエージェントを起動し、次の指示を渡す:

> back-fill された `docs/specs/$CURRENT_FEATURE.md` の受け入れ基準をテストケースに翻訳し、不足分を補ってください。既存テストとの重複は避け、テストランナーを実行してグリーンを確認してください。

### 5. このサイクルの完了報告

以下を簡潔に報告:

1. 今回の `$CURRENT_FEATURE`
2. 変更したファイル一覧
3. 後追いで作成された仕様書のパス (`docs/specs/$CURRENT_FEATURE.md`)
4. 追加したテスト一覧
5. レビューの最終判定
6. 残課題があれば箇条書き

---

## サイクル間の振る舞い（モード継続）

1 件の処理サイクルが終わったら、以下のメッセージを表示して **次の依頼を待つ** こと:

```
【1 件完了 — develop モード継続中】

次の依頼はありますか?
- 次の機能 / 修正内容を教えてください
- 終了する場合は「終了」「exit」「終わり」「もう大丈夫」などと伝えてください
```

ユーザーから次のメッセージが来たら、その内容を **再び「依頼」として解釈** し、「1 件の処理サイクル」を最初（Step 0）から実行してください。

---

## モード終了の判定

ユーザーのメッセージが以下のいずれかに該当する場合、develop モードを **終了** してください:

- 明示的な終了語句: 「終了」「exit」「終わり」「もう大丈夫」「やめる」「ストップ」「stop」「quit」
- 別モードへの切り替えを示唆: 「設計に戻りたい」「別のことしたい」「いったん休憩」
- 他のスラッシュコマンドの実行 (例: `/cc-development-team:design`): この場合は自動的にモードが切り替わる

判断に迷う場合（曖昧な発言）は、必ずユーザーに確認:

```
develop モード中ですが、終了してよろしいですか?
- yes: モード終了
- no: 引き続き依頼をお聞きします
```

## モード終了時の最終報告

モードを抜けるとき、以下をユーザーに報告してください:

1. **このセッションで処理した件数** と各 `$CURRENT_FEATURE` 名
2. 累計で変更したファイル数、追加したテスト数
3. 全件の仕様書パス一覧
4. 「develop モードを終了しました」のアナウンス

---

## モード中の禁止事項

- ユーザーの依頼を受けたのに developer 起動をスキップする
- back-fill フェーズ (architect) を省略する
- ユーザーが終了の意思を明示していないのに勝手にモードを抜ける
- develop モード中に勝手に `/cc-development-team:design` 等を呼ぶ
- 1 件のサイクルが終わってないうちに「次の依頼を教えてください」と聞く

## 注意事項（モード永続化の仕組み）

このモードは **Claude の会話文脈** によって保持されます。以下のケースでは自動的にモードが失われる/中断される可能性があります:

- 長時間の会話で context が圧縮された場合
- 他のスラッシュコマンドを実行した場合
- Claude Code を再起動した場合

その際は、ユーザーが再度 `/cc-development-team:develop` を実行することでモードに戻れます。

---
description: security-reviewer サブエージェントを呼び出してセキュリティ専門点検を実施する
argument-hint: [feature-name | --all]
---

引数: `$ARGUMENTS`

このコマンドは **セキュリティ専門点検** を行います。`reviewer` の品質レビューとは別物で、OWASP Top 10・秘密情報・依存脆弱性・認可漏れなどに集中します。修正は実装しません（指摘のみ）。

## いつ使うか

- 認証・認可・決済・個人情報・ファイルアップロード・外部 API 連携を扱う機能を実装した直後
- リリース前のセキュリティ点検
- 依存ライブラリのアップデート後
- 怪しい挙動・インシデント疑いの調査

---

## Step 0: ターゲット選択ヒアリング

**`$ARGUMENTS` が指定されている場合**:
- `--all` または空文字 → プロジェクト全体を `$TARGET=--all` として保持して Step 1 へ
- それ以外 (feature-name) → その値を `$TARGET` として保持して Step 1 へ

**`$ARGUMENTS` が空の場合**: 以下のヒアリングを実施。

### Step 0-1: 対象範囲の確認

ユーザーに以下を尋ねる:

```
【セキュリティ点検の対象】
番号で答えてください:

a) プロジェクト全体を点検
b) 既存の特定機能を点検
```

### Step 0-2a: a) を選ばれた場合

`$TARGET = "--all"` として保持して Step 1 へ進む。

### Step 0-2b: b) を選ばれた場合

`docs/specs/*.md` を `Glob` で列挙（`_template.md` 除外）。各ファイルを `Read` して以下を抽出:

- **feature-name**: ファイル名から `.md` を除いたもの
- **ステータス**: 末尾の `## ステータス:` 行から
- **概要**: `## 概要` セクションの最初の非コメント非空行（HTML コメントは除外）。約 40 文字でトランケート。記入が無ければ `(概要未記入)`

番号付きリストで提示:

```
既存の仕様書から点検対象を選んでください (番号または feature-name で回答):

1. login-flow      (確定 2026-05-10)   — Firebase Auth でログインする機能
2. book-search     (確定 2026-05-08)   — Google Books API で書籍検索
3. note-add        (back-fill 2026-05-12) — 読んだ本に感想を残す機能
...
```

ユーザーが選んだ feature-name を `$TARGET` として保持。

---

## Step 1: 起動

`security-reviewer` サブエージェントを起動し、次の指示を渡してください:

> セキュリティレビューを実施してください。対象: `$TARGET` (`--all` ならプロジェクト全体、feature-name ならその関連コードに集中)。
> `dept/security-reviewer/CLAUDE.md` のプロジェクト固有要件 (PII・規制・脅威モデル) を踏まえて、OWASP Top 10、秘密情報、依存脆弱性、認可漏れを中心に点検してください。severity 付きの報告書を出力してください。

## Step 2: 結果報告

security-reviewer から受け取った報告を **ユーザーにそのまま提示** してください（要約せず、severity・ファイル行番号・攻撃シナリオ・修正案を残す）。

## Step 3: 修正連携の確認

報告に Critical / High の指摘が含まれる場合、ユーザーに以下を尋ねてください:

```
セキュリティ指摘の修正を developer に依頼しますか?
- yes: 指摘ごとに developer サブエージェントを呼んで修正させます
- no: 一旦報告だけで終わります（ユーザーが手で修正、または後で対応）
```

ユーザーが `yes` の場合のみ、各指摘について `developer` サブエージェントを呼んで修正を依頼します。修正完了後は **再度 `/cc-development-team:security-review <feature-name>` を回して再点検することを推奨** とユーザーに案内してください。

## やってはいけないこと

- ユーザー確認なしに `developer` に修正を投げる
- セキュリティ報告を要約しすぎて本質を消す（severity と攻撃シナリオは必ず残す）
- 「全部 OK」と報告する前に静的検索 (`Grep` / 依存スキャン) をスキップする

「$ARGUMENTS」

このコマンドは **セキュリティ専門点検** を行います。`reviewer` の品質レビューとは別物で、OWASP Top 10・秘密情報・依存脆弱性・認可漏れなどに集中します。修正は実装しません（指摘のみ）。

## いつ使うか

- 認証・認可・決済・個人情報・ファイルアップロード・外部 API 連携を扱う機能を実装した直後
- リリース前のセキュリティ点検
- 依存ライブラリのアップデート後
- 怪しい挙動・インシデント疑いの調査

## 対象範囲の決定

- `$ARGUMENTS` が feature-name (kebab-case) → その機能の関連コードに集中
- `$ARGUMENTS` が `--all` または空 → プロジェクト全体を点検

## Step 1: 起動

`security-reviewer` サブエージェントを起動し、次の指示を渡してください:

> セキュリティレビューを実施してください。対象: `$ARGUMENTS` (空または --all ならプロジェクト全体)。
> `dept/security-reviewer/CLAUDE.md` のプロジェクト固有要件 (PII・規制・脅威モデル) を踏まえて、OWASP Top 10、秘密情報、依存脆弱性、認可漏れを中心に点検してください。severity 付きの報告書を出力してください。

## Step 2: 結果報告

security-reviewer から受け取った報告を **ユーザーにそのまま提示** してください（要約せず、severity・ファイル行番号・攻撃シナリオ・修正案を残す）。

## Step 3: 修正連携の確認

報告に Critical / High の指摘が含まれる場合、ユーザーに以下を尋ねてください:

```
セキュリティ指摘の修正を developer に依頼しますか?
- yes: 指摘ごとに developer サブエージェントを呼んで修正させます
- no: 一旦報告だけで終わります（ユーザーが手で修正、または後で対応）
```

ユーザーが `yes` の場合のみ、各指摘について `developer` サブエージェントを呼んで修正を依頼します。修正完了後は **再度 `/cc-development-team:security-review <feature-name>` を回して再点検することを推奨** とユーザーに案内してください。

## やってはいけないこと

- ユーザー確認なしに `developer` に修正を投げる
- セキュリティ報告を要約しすぎて本質を消す（severity と攻撃シナリオは必ず残す）
- 「全部 OK」と報告する前に静的検索 (`Grep` / 依存スキャン) をスキップする

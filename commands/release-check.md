---
description: リリース前の総合チェック。MVP 完了 / テスト / 手動タスク / 仕様書整合性 / セキュリティ履歴を確認し、リリースノート ドラフトも自動生成する。
---

このコマンドは **公開・デプロイ前の最終確認** を実施します。何も自動で公開・push・タグ付けしません（チェックと案内のみ）。

---

## Step 1: MVP 完了確認

`docs/vision/roadmap.md` を `Read` で読み、`## MVP` セクションを抽出:

- 全件 `- [x]` → ☑ 合格
- `- [ ]` が残っている → ☐ 不合格（未完了一覧を保持）
- `- [-]` (スキップ) があれば理由を確認（roadmap.md にメモがあるか）

`docs/vision/roadmap.md` が無い場合は「vision/roadmap が未作成」として警告（develop のみで進めている可能性、その場合は MVP の判定はスキップ）。

## Step 2: テスト全件実行

テストコマンドの取得:
1. `dept/developer/CLAUDE.md` または `dept/tester/CLAUDE.md` の「テスト実行コマンド」を `Read` で取得
2. 取得できなければプロジェクトから推定（`package.json` の `test` スクリプト、`pytest`、`go test ./...`、`./gradlew test`、`xcodebuild test` 等）

`Bash` で実行:
- 全件グリーン → ☑ 合格
- 失敗あり → ☐ 不合格（失敗一覧を保持）

## Step 3: 手動タスク確認

`docs/manual-tasks/*.md` を `Glob` で列挙（`_template.md` 除外）。各ファイル内で:
- `☐ 未着手` または `☐ 着手中` の件数を集計

- 0 件 → ☑ 合格
- 1 件以上 → ☐ 不合格（未完了一覧を feature-name 付きで保持）

## Step 4: 仕様書とコードの整合性

内部的に `/cc-development-team:sync-spec` 相当の点検を実施（または `sync-spec` コマンドの内容を読み込んで実行）:

各仕様書について実コードとの乖離を集計:
- 乖離 0 件 → ☑ 合格
- 乖離あり → ⚠ 要注意（必ずしも不合格ではないが、リストアップ）

## Step 5: セキュリティ点検履歴

`Bash` で `git log -30 --grep='security\|セキュリティ\|sec-review' --pretty=format:"%ci %s"` を実行:

- 直近 30 日以内にヒット → ☑ 合格
- それより古い / 履歴無し → 以下の条件で分岐:
  - 認証・決済・PII 関連の仕様書 (`docs/specs/` で `auth|login|payment|user|signup|password|profile` を含むファイル名) があれば → ☐ 不合格（実行推奨）
  - 該当機能なし → ⚠ 警告のみ（実行推奨だが致命的ではない）

## Step 6: リリースノート ドラフトの自動生成

`docs/specs/*.md` を全件 `Read` し、各仕様書から以下を抽出:

- ファイル末尾の「ステータス」が `確定` / `back-fill` / `更新` のもの → リリースに含める
- 「概要」セクションを抽出（または「ひとこと」相当の 1〜2 行）
- ステータスが `更新` のものは「改善」セクションへ、それ以外は「新機能」セクションへ

以下のフォーマットで組み立てる（バージョン番号はユーザーに尋ねる）:

```
## v<バージョン> — <アプリ名 (vision.md から)>
リリース日: <YYYY-MM-DD (今日の日付)>

### 新機能
- **<feature-name>**: <概要 1〜2 行>
- ...

### 改善 (該当があれば)
- **<feature-name>**: <概要>

### 既知の制約 / TODO
- <手動タスク未完了などがあれば箇条書き>
```

バージョン番号がまだ無ければユーザーに尋ねる:
```
リリースノートのバージョン番号を教えてください (例: v0.1.0 / v1.0.0-beta.1 など)
```

## Step 7: 総合判定とリリースノート出力

以下のフォーマットで結果を表示:

```
【リリース準備状況】

☑/☐ MVP 完了: <m/n>
☑/☐ 全テスト グリーン: <n/n>
☑/☐ 手動タスク 完了: <0 未完了 / X 未完了>
☑/⚠ 仕様書と実装の整合性: <OK / 乖離 N 件>
☑/☐ セキュリティ点検履歴 (直近 30 日): <あり / 無し>

判定: <☑ リリース可能 / ⚠ 一部要対応 / ☐ 要対応>

────────────────────────
【リリースノート ドラフト】

(上で生成したリリースノート全文)

────────────────────────
【次のアクション】

(判定が ☑ なら) リリース可能です。
- リリースノートを CHANGELOG.md や Release 機能に貼り付け
- git tag <バージョン> でタグ付け（手動で実施）
- お疲れさまでした

(判定が ⚠ や ☐ なら) 以下を解消してから再度 release-check を実行してください:
1. <未完了項目 1 の具体的なアクション>
2. <未完了項目 2 の具体的なアクション>
...
```

## やってはいけないこと

- **何かを自動で公開・push・tag 付け・デプロイする**（チェックと案内のみ。実行はユーザー自身で）
- 不合格項目があるのに「OK」と判定する
- リリースノートを脚色する（仕様書からの抽出に留める、過度な装飾禁止）
- バージョン番号を勝手に決める（必ずユーザーに尋ねる）
- このコマンド実行中にコードや仕様書を書き換える（読み取り専用に近いコマンド）

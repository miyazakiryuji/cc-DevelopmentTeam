`docs/specs/` 配下の仕様書と、実際のコードの **乖離 (drift)** を点検します。

引数: `$1`

- `$1` が指定されていれば `docs/specs/$1.md` のみを対象にする
- 省略時は `docs/specs/*.md` 全件を対象にする

## 手順

1. **対象仕様書を列挙**: 上記ルールに従って対象ファイルを Glob で取得する。1件も無ければ「仕様書が見つかりません。`/cc-development-team:design` または `/cc-development-team:develop` を先に実行してください」と案内して終了する。

2. **各仕様書について以下を実施**:

   a. 仕様書を `Read` で読み込み、以下を抽出する:
      - 設計セクションで言及されているファイル / モジュール / 関数 / API エンドポイント
      - 受け入れ基準
      - データモデル / スキーマ

   b. 抽出した識別子・ファイルパスを `Grep` / `Glob` / `Read` で実コードと照合する:
      - 仕様書に書かれたファイル・関数が **存在するか**
      - 仕様書に書かれた API シグネチャと実装が **一致するか**
      - 受け入れ基準を満たすテストが **どこかに存在するか**

   c. ドリフトを以下のいずれかに分類する:
      - **`仕様あり・実装なし`**: 仕様書には書かれているが、コードに無い
      - **`実装あり・仕様なし`**: コードにあるが、仕様書に書かれていない（または書き古された）
      - **`仕様と実装の差異`**: 名前・シグネチャ・振る舞いが食い違う
      - **`整合`**: 一致している

3. **レポート出力**: 仕様書ごとに、ドリフト件数と詳細を以下の形式でまとめる:

   ```
   ## docs/specs/<name>.md
   - 整合: N 件
   - 仕様あり・実装なし: M 件
     - <該当箇所と仕様書側の記述>
   - 実装あり・仕様なし: K 件
     - <該当ファイル:行番号>
   - 仕様と実装の差異: L 件
     - <該当箇所と差分の要約>
   ```

4. **推奨アクション**: 仕様書ごとに、推奨される次のアクションを 1〜2 行で添える:
   - 大きなドリフト → `/cc-development-team:develop <name>` で back-fill しなおす
   - 小さなドリフト → `architect` に仕様書修正を依頼する
   - 整合 → 何もしなくてよい

## やらないこと

- 勝手にコードや仕様書を書き換える（このコマンドは **報告のみ**）
- 受け入れ基準の意味解釈で迷ったときに憶測で「整合」と判定する（不明は `要確認` として残す）

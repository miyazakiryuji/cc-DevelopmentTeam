---
description: docs/specs/ 配下の仕様書と実コードの整合性をチェックし、ドリフトを報告する。
argument-hint: [feature-name(省略時は全件)]
---

`docs/specs/` 配下の仕様書と、実際のコードの **乖離 (drift)** を点検します。

引数: `$1`

---

## Step 0: ターゲット選択ヒアリング

**`$1` が指定されている場合**: `docs/specs/$1.md` を `$TARGET` として保持して Step 1 へ。

**`$1` が空の場合**: 以下のヒアリングを実施。

### Step 0-1: 対象範囲の確認

ユーザーに以下を尋ねる:

```
【整合性チェックの対象】
番号で答えてください:

a) 全件チェック（docs/specs/ 全部）
b) 既存の特定機能だけチェック
```

### Step 0-2a: a) を選ばれた場合

`$TARGET = "--all"` として保持して Step 1 へ進む。

### Step 0-2b: b) を選ばれた場合

`docs/specs/*.md` を `Glob` で列挙（`_template.md` 除外）。番号付きで提示:

```
点検したい仕様書を選んでください (番号または feature-name で回答):

1. login-flow      (確定 2026-05-10)
2. book-search     (確定 2026-05-08)
...
```

ユーザーが選んだ feature-name を `$TARGET` として保持。

---

## Step 1: 対象仕様書の列挙

- `$TARGET == "--all"` → `docs/specs/*.md` 全件 (`_template.md` 除く)
- それ以外 → `docs/specs/$TARGET.md` のみ

1件も無ければ「仕様書が見つかりません。`/cc-development-team:design` または `/cc-development-team:develop` を先に実行してください」と案内して終了する。

## Step 2: 各仕様書について以下を実施

   a. 仕様書を `Read` で読み込み、以下を抽出する:
      - 設計セクションで言及されているファイル / モジュール / 関数 / API エンドポイント
      - 受け入れ基準
      - データモデル / スキーマ

   b. 抽出した識別子・ファイルパスを `Grep` / `Glob` / `Read` で実コードと照合する:
      - 仕様書に書かれたファイル・関数が **存在するか**
      - 仕様書に書かれた API シグネチャと実装が **一致するか**
      - 受け入れ基準を満たすテストが **どこかに存在するか**

   c. ドリフトを以下のいずれかに分類する:
      - **`仕様あり・実装なし`**: 仕様書には書かれているが、コードに無い
      - **`実装あり・仕様なし`**: コードにあるが、仕様書に書かれていない（または書き古された）
      - **`仕様と実装の差異`**: 名前・シグネチャ・振る舞いが食い違う
      - **`整合`**: 一致している

## Step 3: レポート出力

仕様書ごとに、ドリフト件数と詳細を以下の形式でまとめる:

   ```
   ## docs/specs/<name>.md
   - 整合: N 件
   - 仕様あり・実装なし: M 件
     - <該当箇所と仕様書側の記述>
   - 実装あり・仕様なし: K 件
     - <該当ファイル:行番号>
   - 仕様と実装の差異: L 件
     - <該当箇所と差分の要約>
   ```

## Step 4: 推奨アクション

仕様書ごとに、推奨される次のアクションを 1〜2 行で添える:
   - 大きなドリフト → `/cc-development-team:develop <name>` で back-fill しなおす
   - 小さなドリフト → `architect` に仕様書修正を依頼する
   - 整合 → 何もしなくてよい

## やらないこと

- 勝手にコードや仕様書を書き換える（このコマンドは **報告のみ**）
- 受け入れ基準の意味解釈で迷ったときに憶測で「整合」と判定する（不明は `要確認` として残す）

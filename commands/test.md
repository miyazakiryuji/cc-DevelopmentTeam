`tester` 部署にテストの設計・実装・実行を任せるためのコマンドです。
**結果報告まではこのコマンドが責任を持ちます。改善の自動適用はしません。** 発見した問題の修正は、ユーザーが承認した場合のみ developer に依頼します。

引数: `$1`

---

## Step 0: ターゲット選択ヒアリング

**`$1` が指定されている場合**: `$TARGET = $1` として保持して Step 1 へ。

**`$1` が空の場合**: 以下のヒアリングを実施。

### Step 0-1: 対象範囲の確認

```
【テスト対象】
番号で答えてくださいね!

a) 特定機能のテスト（仕様書から該当機能を選ぶ）
b) プロジェクト全体のテスト実行（既存テストの再実行のみ。新規テスト作成はしない）
```

### Step 0-2a: a) を選ばれた場合

`docs/specs/*.md` を `Glob` で列挙（`_template.md` 除外）。各ファイルを `Read` して以下を抽出:

- **feature-name**: ファイル名から `.md` を除いたもの
- **ステータス**: 末尾の `## ステータス:` 行から
- **概要**: `## 概要` セクションの最初の非コメント非空行（HTML コメントは除外）。約 40 文字でトランケート

番号付きで提示:

```
テスト対象の機能を選んでください (番号または feature-name で回答):

1. login-flow      (確定 2026-05-10)   — Firebase Auth でログインする機能
2. book-search     (確定 2026-05-08)   — Google Books API で書籍検索
...
```

ユーザーが選んだ feature-name を `$TARGET` として保持。

### Step 0-2b: b) を選ばれた場合

`$TARGET = "--all"` として保持して **Step 2 へ直接進む**（新規テスト作成はスキップ、既存テスト実行のみ）。

---

## Step 1: テスト設計・追加 (tester)

**`$TARGET` が特定 feature-name の場合のみ実施**。`--all` の場合はスキップして Step 2 へ。

`tester` サブエージェントを起動し、次の指示を渡す:

> 機能 `$TARGET` のテストを設計・実装してください。
>
> - `docs/specs/$TARGET.md` が存在する場合: 受け入れ基準を Given-When-Then の形でテストに翻訳
> - 存在しない場合: 実コードと `docs/requirements/$TARGET.md`（あれば）から推定して書く。不確実な箇所は `// TODO(spec): ` コメントで明示
>
> 既存テストがあれば命名・構造を揃える。テストファイルの追加・修正のみ行い、**実装コードには手を入れないこと**。テスト実装後は `Step 2` でこのコマンドがテストを実行するので、ここではテスト実行までは行わなくてよい。
>
> 出力:
> - 追加 / 修正したテストファイルのパス
> - 各テストケースの意図（どの受け入れ基準に対応するか）

---

## Step 2: テスト全体実行

`Bash` でプロジェクト全体のテストランナーを実行する。実行コマンドは以下の優先順で決める:

1. `dept/tester/CLAUDE.md` の「テスト実行コマンド」記載
2. `CLAUDE.md`（プロジェクトルート）の「テスト実行コマンド」記載
3. `package.json` の `scripts.test` / `pytest`（`pyproject.toml` / `pytest.ini` 検知）/ `go test ./...` / `cargo test` / `bundle exec rspec` 等を推定

実行結果を保持:

- **合格件数 / 失敗件数 / スキップ件数**
- 失敗テスト一覧（テスト名・対応ファイル・エラーメッセージ要約）
- カバレッジレポートが取れる場合は変化量

---

## Step 3: 結果報告

以下のフォーマットでユーザーに報告:

```
【テスト結果】
対象: <$TARGET or "全体">
新規追加: N 件
合格: A 件 / 失敗: B 件 / スキップ: C 件

# 失敗の概要
- <テスト名>: <エラーメッセージ要約> (該当ファイル:行)
- ...

# 改善候補（tester が気付いた範囲）
- <候補 1>: <なぜ改善が必要か>
- ...
```

「改善候補」には以下のような内容を入れる（tester が Step 1 の作業中に気付いた範囲のみ）:

- 仕様書から導いた受け入れ基準のうち、テストで表現できなかったもの（仕様が曖昧 等）
- 既存テストのフレイキー疑いや重複
- カバレッジが薄い領域

---

## Step 4: 改善の適用確認

**失敗テスト or 改善候補が 1 件以上ある場合のみ**、以下をユーザーに尋ねる:

```
発見された問題があります。修正に進みますか?

a) 失敗テストを developer に修正させる（実装側のバグ修正）
b) 改善候補を反映する（テスト追加 / 仕様書修正等。具体内容は次に確認します）
c) このまま終了（報告のみ）

(複数選んでも OK です。番号や a/b/c で答えてください!)
```

### Step 4-a: a) 失敗テストの修正

失敗テスト一覧をユーザーと確認してから、`developer` サブエージェントに修正依頼:

> 以下のテストが失敗しています。テストが期待する振る舞いに合うよう **実装側を修正** してください。テスト側は書き換えないこと（テストが間違っていると判断した場合は **実装を変えず**、報告にとどめてユーザーに差し戻すこと）。
>
> 失敗テスト:
> - <テスト名>: <エラー要約>
> - ...

developer の修正完了後、**Step 2 のテスト実行を再度実施** して全件グリーンになったか確認。グリーンになるまで Step 4-a を反復（ただし 3 回失敗したら一度立ち止まってユーザーに状況報告）。

### Step 4-b: b) 改善候補の反映

改善候補を 1 件ずつユーザーに見せ、それぞれ「適用する / しない」を確認。適用するものについては:

- テスト追加が必要 → `tester` を再度呼んで追加
- 仕様書修正が必要 → `architect` を呼んで該当箇所を更新（破壊的変更はユーザー承認の上で）

### Step 4-c: c) このまま終了

報告だけ残してコマンド終了。

---

## Step 5: 完了報告

以下を簡潔に出力:

1. 今回の対象 (`$TARGET`)
2. 追加 / 修正したテストファイル一覧
3. 最終テスト結果（合格 / 失敗 / スキップ）
4. 適用した修正の内容（Step 4 で実施した場合のみ）
5. 残課題（未適用の改善候補・依然失敗しているテスト 等）

---

## やってはいけないこと

- **失敗テストや改善候補をユーザー確認なしに修正する**（このコマンドの一番大事なルール）
- テスト通過のために実装側を勝手に書き換える（必ず Step 4-a 経由）
- `dept/tester/CLAUDE.md` に「モック禁止」と明記されているのに安易にモックする
- カバレッジ稼ぎだけの無意味なテストを足す
- 仕様外のシナリオを勝手にテストして仕様化してしまう

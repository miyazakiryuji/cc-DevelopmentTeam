機能名: `$1`

以下の4部署フローを **この順序で** 実行してください。各フェーズで使うサブエージェントは、このプラグインが提供している `architect` / `developer` / `reviewer` / `tester` です。

## フェーズ 1: 設計 (architect)

`architect` サブエージェントを起動し、次の指示を渡してください:

> 機能 `$1` の仕様書を `docs/specs/$1.md` に作成してください。受け入れ基準まで明文化し、開発部署が一意に実装できる粒度にしてください。既存コードを十分に調査し、影響範囲を列挙してください。

`docs/specs/$1.md` が存在しない場合は新規作成、存在する場合は改訂履歴を末尾に追記して更新します。

設計完了後、生成された仕様書の要旨をユーザーに提示し、**実装に進んでよいかを確認**してください。

## フェーズ 2: 実装 (developer)

ユーザー承認後、`developer` サブエージェントを起動し、次の指示を渡してください:

> `docs/specs/$1.md` に基づいて実装してください。最小差分で、CLAUDE.md と既存慣習に沿ってください。ビルドと既存テストが緑のまま完了させてください。

## フェーズ 3: レビュー (reviewer)

実装完了後、`reviewer` サブエージェントを起動し、次の指示を渡してください:

> `docs/specs/$1.md` と実装差分の整合性、品質、セキュリティを点検し、severity 付きで指摘してください。最後に Approve / Request Changes / Block の判定を出してください。

`Request Changes` 以上の指摘があれば、developer に差し戻してフェーズ 2 へ戻ります。

## フェーズ 4: テスト (tester)

レビュー Approve 後、`tester` サブエージェントを起動し、次の指示を渡してください:

> `docs/specs/$1.md` の受け入れ基準をテストケースに翻訳し、適切なレイヤで追加してください。テストランナーを実行してグリーンを確認してください。

## 完了報告

すべてのフェーズが完了したら、ユーザーに以下を簡潔に報告してください:

1. 仕様書のパス (`docs/specs/$1.md`)
2. 変更したファイル一覧
3. 追加したテスト一覧
4. レビューの最終判定
5. 残課題があれば箇条書き

## 1. 目的

ユーザーが **Google アカウントで認証** して Sharedo を使い始められる状態を作る。
新規アカウント発行のフォーム入力を避け、**最短でリスト操作画面に到達** させることで、
家族・友人を巻き込むハードルを下げる。

## 2. 対象ユーザー

- 主: 同居家族 (夫婦) で家事 / 買い物リストを共有したい人
- 副: 友人グループ (旅行計画) でリスト共有したい人
- いずれも **Google アカウントを持っている** ことを前提とする (持っていない人は対象外、Phase 2 でメール認証検討)

## 3. 業務フロー / ユーザージャーニー

### 3.1 ハッピーパス (新規ユーザー)

1. ユーザーが Sharedo の URL にアクセス
2. ランディングページが表示され、「Google でログイン」ボタンを見つける
3. ボタン押下で Google OAuth 同意画面に遷移
4. Google アカウントを選び、Sharedo へのアクセスを許可
5. Sharedo のリスト一覧画面 (`/lists`) に着地
6. リスト 0 件の状態で「最初のリストを作りましょう」CTA が表示される

### 3.2 ハッピーパス (既存ユーザー)

1. ユーザーが Sharedo の URL にアクセス
2. ブラウザに残ったセッション Cookie で自動ログイン
3. リスト一覧画面 (`/lists`) に直接着地

### 3.3 代替パス (招待リンク経由の新規ユーザー)

1. 別のユーザーから招待 URL (`/invite/<token>`) を受け取る
2. URL を開くと、未ログインのためまずログインを促される
3. Google OAuth でログイン完了後、招待されたリストに自動で参加し `/lists/<id>` に着地

### 3.4 失敗パス

- Google OAuth 認証中にキャンセル → ランディングに戻る + 「ログインがキャンセルされました」表示
- セッション期限切れ → 任意ページにアクセスすると `/login` にリダイレクト
- Google アカウント取得失敗 (ネットワーク等) → ランディングに戻る + 「ログインに失敗しました。時間をおいて再度お試しください」表示

## 4. 機能の構成要素

| サブ ID | 構成要素 | 役割 |
| --- | --- | --- |
| F-001-1 | ランディングページ | 未ログイン時の入口。CTA = Google ログイン |
| F-001-2 | Google OAuth 連携 | Supabase Auth 経由で Google OAuth フロー |
| F-001-3 | セッション管理 | ログイン状態を Cookie で永続化 |
| F-001-4 | プロフィール初期化 | 初回ログイン時に `users` テーブルへレコード作成 |
| F-001-5 | ログアウト | セッション破棄 |
| F-001-6 | 認可ガード | `/lists` 配下は未ログインで `/login` リダイレクト |

## 5. 機能固有の非機能要件

- **性能:** ログインボタン押下から `/lists` 着地まで **3 秒以内** (Google 側応答待ち除く)
- **セキュリティ:**
  - PKCE フロー必須 (Supabase デフォルト)
  - State パラメータで CSRF 対策
  - 取得した token は **Cookie に HttpOnly + Secure + SameSite=Lax** で保存
  - クライアントサイド JS から token にアクセスさせない
- **アクセシビリティ:** ログインボタンはキーボード操作可能、スクリーンリーダー対応の aria-label
- **可用性:** Supabase Auth がダウンしている場合、ユーザーフレンドリーなエラー表示 (Supabase 直のスタックトレースを見せない)

## 6. スコープ (含むもの・含まないもの)

### 6.1 含むもの (MVP)

- Google OAuth でのログイン / ログアウト
- セッション維持 (Cookie ベース)
- 初回ログイン時のユーザーレコード作成
- 認可ガード (未ログインで保護ページにアクセス時のリダイレクト)

### 6.2 含まないもの (Phase 2 以降)

- メール / パスワード認証
- 他の OAuth プロバイダ (Apple / GitHub)
- 2 要素認証
- アカウント削除フロー (まずは support 経由で対応)
- パスワード変更 / プロフィール編集 (`/settings` で別途)

## 7. 想定リスク

| リスク | 影響度 | 対策 |
| --- | --- | --- |
| Google OAuth クライアント ID 発行漏れ | High | 開発前に手動タスクとして追跡 ([manual-tasks](../manual-tasks/login-flow.md)) |
| Supabase Auth の料金プラン上限 | Medium | MVP 想定規模 (< 1000 MAU) では Free プランで十分。要モニタリング |
| Google アカウント無いユーザーの離脱 | Medium | MVP では割り切る。Phase 2 でメール認証導入を検討 |
| セッション固定攻撃 | Low | Supabase デフォルトの実装で対応済み |
